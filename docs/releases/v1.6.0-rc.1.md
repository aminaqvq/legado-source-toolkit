# v1.6.0-rc.1 — Batch Deep Validate

## Highlights

v1.6 brings v1.5's Single Source Lab chain validation (`search → bookInfo → toc → content`) into the batch processing pipeline. You can now validate dozens or hundreds of book sources at three depth levels: fast, standard, and deep — with structured per-source results and aggregate reporting.

## Added

- **Batch validation modes** (`--validate-mode fast | standard | deep`)
  - `fast`: no rule engine — reuses existing structure + connectivity + lightweight search results
  - `standard`: search → bookInfo → toc (stops before content; 3 HTTP requests per source)
  - `deep`: search → bookInfo → toc → content (full chain; 4 HTTP requests per source)
- **Status classification** per source: PASS / PARTIAL_PASS / FAIL / BLOCKED / NEEDS_LOGIN / UNSUPPORTED / RISKY / UNKNOWN
- **Batch validation summary** (`BatchValidationSummary`) with aggregation by failure reason, host, group, and source type
- **`--batch-concurrency`** CLI/API parameter (default 8) for controlling batch validation parallelism
- **`maxStage` parameter** on `verifyAllRules()` for stopping chain at a specific stage
- **19 new tests** covering status summarization, aggregate counting, mode behavior, chain breaks, and report output

## Changed

- `SourceAnalysis` extended with `batchValidationMode`, `batchValidationStatus`, `batchFailureReasons`, `batchWarnings`, `batchSuggestions`, `batchDurationMs`, `firstFailureStage`
- `ProcessSummary` extended with optional `batchValidation?: BatchValidationSummary`
- `ProcessOptions` extended with `validateMode?: BatchValidationMode` and `batchConcurrency?: number`
- `onProgress` callback now accepts `'batch'` label for UI progress tracking
- `processSources()` pipeline adds a new guarded phase (Phase 6b) after scoring when `validateMode` is set
- CLI `process` command adds `--validate-mode` and `--batch-concurrency` options (both with no default change to existing behavior)

## Security

- All v1.5 security boundaries retained: SSRF protection, redirect re-check, dangerous header filtering
- Batch concurrency default set to 8 (lower than `process --online`'s 16) to limit target site impact
- Response body previews truncated in HTML reports (max 500 chars)
- No Cookie / Authorization headers written to report outputs
- `maxStage` parameter prevents accidental content fetching in standard mode

## Reports

- **JSON** (`reports/sources.json`): new batch fields auto-serialized per source; `reports/summary.json` includes `batchValidation` when present
- **CSV** (`reports/sources.csv`): 6 new columns — validationMode, finalStatus, firstFailureStage, failureReasons, warnings, durationMs
- **HTML** (`reports/report.html`): new "Batch Validation Results" section with summary cards, top-10 failure reason distribution; source tables conditionally show validation columns

## Web UI

- **ResultsPage**: batch validation summary cards and collapsible failure reason distribution when batch data present
- **SourcesPage / SourceTable**: conditional columns for validation mode, final status, and first failure stage

## Tests

- `mapStagesToBatchResult`: 7 status summarization scenarios (PASS, PARTIAL_PASS, FAIL, BLOCKED, UNSUPPORTED, NEEDS_LOGIN, firstFailureStage)
- `summarizeBatchValidation`: aggregate counting with byFailureReason / byHost / byGroup / bySourceType verification
- `runBatchValidation`: standard mode verifies no content fetch; deep mode verifies content fetch
- `verifyAllRules maxStage`: toc stops before content, content runs all 4
- Chain break tests: book_url_missing stops after search; chapter_url_missing stops after toc
- Report serialization: JSON fields, CSV columns, HTML section presence

## Notes

> CLI/API 是启用 v1.6 batch validation 的主要入口；Web UI 当前主要展示 batch validation 结果摘要和 source table 状态列。
>
> ⚠️ batch validation 建议与 `--online` 搭配使用。如果不启用 `--online`，且未设置 `--include-unknown`，unknown availability 源可能会在 batch validation 前被过滤，导致 batch target 为 0。

## Known Limitations

| Limitation | Detail |
|------------|--------|
| Browser Runner | Not supported |
| Android Runner | Not supported |
| Rhino compatibility | Not supported; Node vm sandbox only |
| `java.ajax` | Not executed |
| Login / Cookie / Session | Not handled |
| Cloudflare / CAPTCHA bypass | Not attempted |
| `webView:true` | Not supported |
| Per-source stage details in batch reports | Not yet — use Debug page or P1 enhancement |
| Full stageResults in JSON/CSV/HTML | Summary fields only; full detail deferred |
